一条设备命令的一生

从 desk.move_to_height 出发，理解命令身份、持久化、ACK、物理副作用、验证和恢复为什么必须分开。

约 15 分钟 交互故障模拟 目标：正确处理未知结果
本节的唯一胜利条件：当命令超时或 ACK 丢失时，你不会条件反射地 retry，而会先判断物理副作用是否已经发生。

1. 先认识命令信封

设备命令不是只有函数名和参数。它必须携带足以判断“这是谁、还能不能执行、重复了怎么办”的运行语义。

{
  "commandId": "cmd_desk_0042",
  "taskRunId": "run_focus_0017",
  "capabilityId": "desk.move_to_height",
  "input": { "heightMm": 760 },
  "expectedStateVersion": 108,
  "idempotencyKey": "run_focus_0017:desk:step_02",
  "expiresAt": "2026-07-11T10:15:00Z",
  "policyGrantId": "grant_once_009"
}
字段回答的问题缺失后的风险
commandId我正在跟踪哪一次命令？日志、进度与结果无法关联
idempotencyKey这是不是同一个业务意图的重投？超时重试导致重复移动
expectedStateVersion我的计划基于哪个状态？用旧前提覆盖新状态
expiresAt这项授权现在还有效吗？离线积压命令晚到后突然执行
policyGrantId谁依据哪条规则允许了它？动作无法审计或越过审批

2. 四种“确认”不是一回事

01

Transport ACK

消息到达某个网络端点。只证明“收到了字节”。

02

Accepted

本地校验通过，并在副作用前持久化命令。

03

Reported

执行器或 adapter 上报动作完成。

04

Verified

重新读取状态，确认真实结果符合目标。

只有 Verified 才能支撑业务成功。ACK 可能丢失，执行器上报也可能基于错误或陈旧状态；因此“工具返回 success”不等于“桌子真的到了 760 mm”。

3. 正常生命周期

  1. Proposed：TaskSpec 产生语义动作。
  2. Authorized：Policy 返回 grant 或等待审批。
  3. Persisted：station 在动作前写入 command journal。
  4. Accepted：版本、期限和参数校验通过。
  5. Executing：执行器运动并持续发 progress。
  6. Reported：adapter 上报 terminal result。
  7. Verifying：读取新的 WorkstationSnapshot。
  8. Completed：观察值满足 verification spec。

ROS 2 把这种长动作建模为 Action：发送 goal、接收 feedback、支持 cancel，并最终返回 result。这也是我们未来 ROS adapter 的映射基础。参见 ROS 2 官方接口比较。

    5. 为什么 journal 要先于副作用

    如果 station 先移动桌子、后记录命令，进程可能在两者之间崩溃。重启后系统甚至不知道曾经收到过这条命令。

    校验命令→写入 journal→执行动作→追加结果

    但是“先写 journal”仍然不能消灭所有不确定性：进程可能在物理动作完成后、写入成功结果前崩溃。所以恢复时仍要查询设备状态并 reconciliation。

    6. 幂等不是“请求只来一次”

    幂等契约允许同一业务请求重复传输，而不产生第二次副作用。AWS 的工程文章特别强调：安全 retry 的前提，是服务能够用 caller-provided request identity 识别重复意图。建议阅读 Making retries safe with idempotent APIs。

    收到的请求Station 行为
    新 key + 合法 input持久化并执行
    旧 key + 相同 input返回既有状态或结果，不重新执行
    旧 key + 不同 input返回 identity conflict，绝不猜测
    新 key + 过期 grant拒绝执行

    Cloudflare Workflows 的官方规则同样要求可重试 step 尽量幂等，因为基础设施可能重启并重新执行当前 step。参见 Rules of Workflows。

    8. 带走这三句话

    接受不等于完成。

    Accepted 只说明本地愿意执行。

    上报不等于事实。

    Reported 后仍要 read-after-write verification。

    未知先查再重试。

    Reconciliation 用物理事实关闭不确定性。
