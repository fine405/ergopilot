ErgoPilot · Lesson 01

从前端工程师到 Agent Runtime

先不背框架名。用你已经熟悉的组件状态、API、mutation 和事件流，建立一张能支撑整个项目的心智地图。

约 15 分钟 不要求 Rust 基础 目标：能解释四层架构
本节的唯一胜利条件：看到一个需求时，你能判断它属于认知规划、可靠编排还是设备执行，而不是把所有问题都归给“大模型”。

1. 你已经掌握了一半

Agent 系统不是与前端完全不同的世界。真正的新东西，是副作用更昂贵、运行时间更长、状态来源更多，而且失败后不能简单刷新页面。

熟悉的前端概念Agent / 设备系统中的对应物关键差异
useMutation()设备命令副作用可能已发生，超时后不能盲目重试
TypeScript API schemaTool / Capability schema还要描述风险、前置条件、取消与验证
Redux reducer状态机状态必须持久化，并能从崩溃中恢复
SSE / WebSocket工具事件与设备遥测消息可能延迟、重复、乱序或断线
React Query cache云端设备快照它只是观测值，不是物理世界的真相
Optimistic UI预测设备结果涉及机械动作时通常必须执行后再验证
最重要的认知转换：前端里“请求超时”常常意味着再发一次；物理世界里，它可能意味着桌子已经移动，但成功 ACK 丢了。此时要先读真实状态，而不是再移动一次。

2. Agent 到底是什么

在这个项目里，可以把 Agent 理解成下面这组能力，而不是一个聊天框：

模型+上下文+工具+循环
状态+权限+恢复+评测

前四项让它“能做事”，后四项让它“可以进入真实产品”。

Agent：处理开放性

理解“肩膀有点累、接下来要专注”这种不精确表达，选择合适能力并生成结构化 TaskSpec。

Workflow：处理确定性

按明确步骤检查权限、等待审批、发送命令、超时、恢复，并保证任务最终进入可解释的状态。

所以 ErgoPilot 的核心不是“Agent 还是 Workflow 二选一”，而是让 Agent 负责开放性，让 Workflow 和本地状态机负责确定性。

3. 四层系统地图

01

体验层

TanStack Start、shadcn/ui、AI Elements。输入目标，展示计划、审批、进度与证据。

02

认知层

Mastra、LLM、Tools、MCP。理解意图并产出受 schema 约束的计划。

03

控制层

Hono、Cloudflare Workflows、Durable Objects。持久任务、审批、重试和实时协调。

04

设备层

Tauri、Rust、Tokio、SQLite、Simulator。拥有真实状态并安全执行设备命令。

  1. 体验层收到“准备 90 分钟专注工作”。
  2. 认知层将它转换成 TaskSpec，不输出电机参数。
  3. 控制层检查策略、等待审批并可靠投递命令。
  4. 设备层验证当前状态、执行、汇报进度并再次读取状态。

4. 六个必须先懂的概念

概念先这样理解在 ErgoPilot 中
Tool模型能调用的、有输入输出 schema 的函数prepare_focus_session
Capability系统或设备提供的语义能力，并携带风险与运行语义desk.move_to_height
MCP让 AI 应用发现和调用工具/资源的标准协议向其他 Agent 暴露工位能力
State machine允许哪些状态以及它们之间如何迁移executing → verifying → completed
Idempotency同一命令重复到达，副作用只生效一次ACK 丢失后重投不会再次升桌
Reconciliation不相信猜测，重新读取事实后决定下一步超时后查询桌子真实高度

MCP 只规定“如何发现和调用”，不会自动提供权限、状态恢复或设备安全。官方架构把它定义为 host/client/server 协议，并区分 tools、resources 和 prompts。建议把 MCP Architecture 作为本节的主阅读。

技术解决什么当前深度

6. 三道职责分类

三个选项都保持四个汉字。先凭直觉作答，立即看反馈。

7. 带走这四句话

Agent 处理开放性。

理解意图、补全信息、提出结构化计划。

Workflow 处理确定性。

状态、审批、等待、超时和恢复必须明确。

设备运行时拥有物理事实。

云端看到的是快照，本地读取的才是设备状态。

超时先 reconcile。

物理副作用不确定时，先查事实再决定是否重试。
