副作用前检查

身份

key 是否已存在？相同 key 的 input hash 是否一致？

权限

grant 是否存在、匹配 capability 且未过期？

状态

expected version 是否仍是当前版本？传感器是否新鲜？

约束

输入是否在硬限制内？前置条件是否成立？

持久化

command 是否已在动作前写入 journal？

可恢复性

超时、取消、重启后应该进入哪个状态？

超时决策表

重新观察的事实下一步
目标已经达到记录 verified success，不再执行
状态完全未变化在预算内用同一 identity 重投
状态部分变化停止、suspend，按设备规则补偿或人工处理
设备状态不可读保持 unknown/suspended，不盲目重试
状态版本已变化重新规划，不覆盖新状态

恢复时扫描什么

  1. 加载所有非 terminal command。
  2. 按 command identity 查询既有结果。
  3. 读取设备当前状态。
  4. 将事实与 target / verification spec 比较。
  5. 更新为 succeeded、retryable、suspended 或 failed。
  6. 将恢复结论发送给高层 Workflow。

面试版回答

“设备命令超时不代表动作没发生。我会给命令稳定的 idempotency key，在本地副作用前持久化；如果结果未知，先读取真实设备状态进行 reconciliation。目标已达到就补记成功，未执行才用相同 identity 有界重投，部分执行或不可观测则进入 suspended，而不是盲目 retry。”
